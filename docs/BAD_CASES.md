# Bad Case 复盘

> 目标：让项目不只是“跑通 Demo”，而是能说明问题如何被发现、定位和修复。

## 1. 退款进度回答不够直接

- 问题：`钱咋还没退回来啊，我都等半天了`
- 现象：检索命中“退款进度”，但生成回复可能先说安抚话，或者没有第一句直接解释到账受支付渠道影响。
- 原因定位：retrieval 正确，问题主要在 answer rendering；模型原始回复不稳定。
- 修改方案：增加 answer composer，将 primary evidence 拆成 conclusion、action、caveat，要求第一句直接回答。
- 当前结果：最新报告中该 case 的 Top1 intent 为“退款进度”，reply 能覆盖“支付渠道、订单详情页、处理延迟”。

## 2. 优惠券退款后能否继续使用

- 问题：退款或取消后，用户追问红包/优惠券是否还能继续用。
- 现象：回答容易停留在“以页面为准”，没有区分红包类型、退回条件和查看入口。
- 原因定位：证据中存在条件限制，普通 prompt 容易漏掉限制条件。
- 修改方案：在回复要求中强调“必须覆盖条件限制和具体入口”，并把该类问题加入 grounding 评测。
- 当前结果：仍适合保留为 manual review 类型，后续需要补充更细的优惠券知识条目。

## 3. 发票申请入口不明确

- 问题：用户问“发票在哪里开/怎么申请”。
- 现象：回答可能只说“在订单中申请”，缺少入口路径。
- 原因定位：知识条目和 prompt 对“在哪里看”类问题的约束不够强。
- 修改方案：在 prompt 规则中增加“用户问在哪里看时，第一句必须给具体入口路径”。
- 当前结果：该类问题已进入评测报告，后续应补充真实平台式入口表述。

## 4. 地址写错且骑手已到原地址

- 问题：`地址写搓了，骑手快到老地址，我人不在那咋办`
- 现象：需要同时处理错别字、多轮语义和配送状态，容易混入“未开始配送可修改地址”的旧条件。
- 原因定位：Top1 需要命中“地址修改追问”，辅助证据只能补充流程，不能覆盖主证据。
- 修改方案：Context Builder 区分 primary evidence 和 supporting evidence，并限制辅助证据只补充流程。
- 当前结果：最新报告中 Top1 intent 为“地址修改追问”，回复覆盖“联系骑手、超范围可能无法改送、联系平台客服协商”。

## 5. 食品安全和私下交易高风险问题

- 问题：用户涉及食品安全投诉、私下转账、隐私信息、绕过平台沟通。
- 现象：模型可能为了“帮用户解决问题”给出不合规建议。
- 原因定位：单靠生成模型无法稳定遵守安全边界。
- 修改方案：增加 reply rules，对高风险场景做规则级兜底，禁止承诺赔付、退款成功或引导私下交易。
- 当前结果：评测报告保留 `forbidden_keyword_hits` 字段，用于检查是否出现高风险词。

## 后续修复顺序

1. 先扩展评测集，不急着调模型。
2. 对每个失败 case 标注失败层：retrieval、context builder、generation、reply rules、judge。
3. 每次只改一层，保留修改前后报告。
4. 将能复现的问题沉淀为固定测试或评测样本。
